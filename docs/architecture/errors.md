# Error handling strategy

> How Harmonia crates define, propagate, and log errors across subsystem boundaries.
> Error enums listed here map 1:1 to the crate inventory in [cargo.md](cargo.md).
> Patterns mandated by [standards/RUST.md](../../standards/RUST.md) and the GreptimeDB reference architecture.

## Purpose

Every crate in the workspace owns exactly one error enum. Errors propagate with location tracking and cross subsystem boundaries via `Into` conversions auto-generated by snafu. This document specifies the pattern, the `source` vs `error` naming convention, the boundary propagation strategy, the API boundary response format, and the logging rule. All patterns follow `standards/RUST.md` mandates and the GreptimeDB multi-crate error organization pattern.

---

## The pattern: one error enum per crate

Each of the 14 library crates defines exactly one error enum using `#[derive(Debug, Snafu)]`. That enum is the crate's public error surface; callers only ever see one error type per crate, regardless of how many internal operations can fail.

**Template:**

```rust
use snafu::{ResultExt, Snafu, Location};

#[derive(Debug, Snafu)]
pub enum {CrateName}Error {
    #[snafu(display("{description}: {source}"))]
    VariantName {
        description: String,
        source: SomeExternalError,
        #[snafu(implicit)]
        location: Location,
    },
}
```

**`#[snafu(implicit)]` on `location`:** snafu 0.8 automatically captures the call site's file, line, and column when the variant is constructed via `.context(VariantSnafu { ... })?`. No manual `Location::new()` call is needed.

**`ResultExt`:** The `.context()` method, which snafu generates on `Result<T, E>`, is what attaches the location and additional context fields to the underlying error. Always import `ResultExt` in files that use `.context()`.

---

## `Source` vs `error` field convention

This is the GreptimeDB naming convention. The field name controls whether the error chain walker continues or stops.

**`source` field:** Wraps an error from another Harmonia crate or a well-known external crate (reqwest, sqlx, jsonwebtoken). snafu's chain walker **continues** through `source` fields, so diagnostic tools and log formatters that walk the chain will see the full error ancestry. Use when the wrapped error provides meaningful diagnostic context.

**`error` field:** Wraps an opaque external error, typically serialized to `String`. The chain walker **stops** at `error` fields. Use when the error comes from an external service response or a third-party crate whose error type is unstable, API-defined, or not worth walking into.

**Side-by-side examples from the `zetesis` crate:**

```rust
#[derive(Debug, Snafu)]
pub enum ZetesisError {
    // 'source' field — internal error, walk the chain
    // reqwest::Error has structured context worth preserving
    #[snafu(display("indexer search failed for query {query}: {source}"))]
    IndexerSearch {
        query: String,
        source: reqwest::Error,
        #[snafu(implicit)]
        location: Location,
    },

    // 'error' field — external error, stop walking
    // Cloudflare bypass failure comes from an HTTP response body: opaque string
    #[snafu(display("Cloudflare bypass failed: {error}"))]
    CloudflareBypass {
        error: String,
        #[snafu(implicit)]
        location: Location,
    },
}
```

**The critical distinction:** When a caller logs or displays the error chain, it walks `source` fields recursively. If `CloudflareBypass` used `source: some_bypass_crate::Error` instead of `error: String`, the chain walker would attempt to walk into that external crate's error internals, which are unstable, often opaque, and add noise. The `error: String` field captures the meaningful message at the point where it was received, then stops.

## Cross-subsystem error propagation

When subsystem A calls subsystem B, and B returns an error, A wraps it in a variant of A's own error enum. snafu auto-generates the `From<B::Error>` impl, so `.context(VariantSnafu { ... })?` is all that is needed at the call site.

**Concrete example: Episkope wrapping a ZetesisError:**

```rust
// In crates/episkope/src/error.rs
use snafu::{ResultExt, Snafu, Location};

#[derive(Debug, Snafu)]
pub enum EpiskopError {
    #[snafu(display("search failed: {source}"))]
    SearchFailed {
        source: zetesis::ZetesisError,  // 'source' — walk the chain into zetesis
        #[snafu(implicit)]
        location: Location,
    },
    // ... other Episkope-owned variants
}

// In crates/episkope/src/service.rs
fn trigger_search(query: &str) -> Result<Vec<SearchResult>, EpiskopError> {
    zetesis::search(query).context(SearchFailedSnafu)?
}
```

The `From<ZetesisError>` impl is auto-generated by snafu. The error chain now reads: `EpiskopError::SearchFailed` → `ZetesisError::IndexerSearch` → `reqwest::Error`. Every layer added context (caller location, query string, HTTP details).

**Rule at the trait boundary:** Every public async method that can fail must return `Result<T, {CrateName}Error>`. No `Box<dyn Error>`, no `anyhow::Error` in library crates. The error type is part of the public contract.

## Error at the API boundary

When errors reach `archon` (the HTTP layer), they are converted to structured JSON responses. Internal error chains are never exposed to API clients; that is both a security risk (leaking internal paths, query structures, subsystem names) and a usability problem.

**Conversion approach:** `archon` implements `axum::response::IntoResponse` on a unified `ApiError` type that wraps the subsystem-specific errors.

**Response format:**

```json
{
  "error": "human-readable message appropriate for the client",
  "code": "MACHINE_READABLE_CODE",
  "correlation_id": "uuid-v4-for-log-correlation"
}
```

**HTTP status code mapping:**

| Error class | HTTP status |
|-------------|-------------|
| `NotFound` variants | 404 Not Found |
| `Unauthorized` / `TokenInvalid` / `TokenExpired` | 401 Unauthorized |
| `Forbidden` / permission denied | 403 Forbidden |
| `ValidationFailed` / bad input | 422 Unprocessable Entity |
| All other internal errors | 500 Internal Server Error |

**The correlation ID** is generated at the start of each request (middleware layer in archon) and injected into the tracing span. The same ID appears in the error response and in the log entry, enabling a user report to be matched to a log line.

## The logging rule

**Log errors where HANDLED, not where they occur.**

An error occurs at the call site where it originates. It is handled at the point where the caller makes a decision about what to do: retry, return a fallback value, abort the request, or propagate further. Only the handler logs.

**Examples:**

- Episkope calls Zetesis, gets a `ZetesisError::IndexerSearch`. Episkope decides to retry with a different indexer. Episkope logs at the retry point; Zetesis does not log.
- The retry also fails. Episkope propagates `EpiskopError::SearchFailed` up to archon. archon handles it by returning a 500 response. archon logs once; neither Episkope nor Zetesis logged earlier.
- Multiple log entries for the same error chain is noise. One log entry per error, at the final handling point.

**Log format using tracing:**

```rust
tracing::error!(
    error = %e,
    correlation_id = %correlation_id,
    "search request failed after all retries"
);
```

For errors with location tracking, the location is captured in the error struct itself (via `#[snafu(implicit)]`) and available as `e.location()` if the formatter supports it. Do not repeat the location in the log message; the structured fields are machine-readable.

## Anti-patterns

**No `unwrap()` in library code.** Mandated by `standards/RUST.md`. All fallible operations propagate via `?` with `.context()`. The only acceptable use of `unwrap()` is in test code for setup operations that truly cannot fail in a controlled environment.

**No `Box<dyn Error>`.** Type-safe snafu enums replace `Box<dyn Error>`. The error type is part of the public contract; erasure defeats chain walking, structured logging, and caller-side error matching.

**No `anyhow` in library crates.** `anyhow::Error` is acceptable in `archon` as an application-level escape hatch for startup failures, but never in any of the 14 library crates. Library code must have typed, inspectable errors.

**No bare `?` without `.context()`.** Every `?` in library code must go through `.context(VariantSnafu { ... })`. The bare `?` would discard location information and skip the variant-level context. snafu's `#[snafu(transparent)]` is the only exception; it is explicit and intentional.

**No logging at error origination.** Logging at the call site where an error is created means every intermediate propagation layer adds noise. Log once, at the handling site.

**No `thiserror` in new crates.** The `thiserror` crate may appear in pre-existing Akouo code. New Mouseion crates use `snafu` only. `thiserror` lacks location tracking and the `source`/`error` chain-walking convention.
