# Quality profiles

> Scoring and upgrade logic for the Want/Release/Have lifecycle.
> Subsystem ownership: [subsystems.md](../architecture/subsystems.md). Kritike owns quality profile enforcement; Episkope checks profiles when evaluating candidate releases.

## Purpose

Quality profiles are the scoring system that determines whether a release is acceptable for download and whether an existing have should be upgraded. A profile pairs a minimum acceptable score (the floor: reject releases below this) with an upgrade-until score (the ceiling: stop searching for upgrades once the have meets or exceeds this). Kritike enforces profiles at assessment time: when a new import arrives, Kritike compares its quality score against the active profile. Episkope checks profiles when evaluating candidate releases found by Zetesis: a release must meet `min_quality_score` before being handed to Syntaxis for download.

System-wide profiles replace the manual upgrade decisions required by earlier *arr tooling. The user defines named profiles; the system handles the rest.

## `Quality_profiles` table

```sql
CREATE TABLE quality_profiles (
    id                         INTEGER PRIMARY KEY,
    name                       TEXT NOT NULL UNIQUE,
    media_type                 TEXT NOT NULL CHECK(media_type IN (
                                   'music', 'audiobook', 'book', 'comic',
                                   'podcast', 'movie', 'tv'
                               )),
    min_quality_score          INTEGER NOT NULL,
    upgrade_until_score        INTEGER NOT NULL,
    min_custom_format_score    INTEGER NOT NULL DEFAULT 0,
    upgrade_until_format_score INTEGER NOT NULL DEFAULT 0,
    upgrades_allowed           INTEGER NOT NULL DEFAULT 1
);

CREATE INDEX idx_quality_profiles_media_type ON quality_profiles(media_type);
```

| Column | Type | Notes |
|--------|------|-------|
| `id` | `INTEGER PRIMARY KEY` | Auto-increment. Profiles are admin-managed, low cardinality; integer ID is correct here. |
| `name` | `TEXT NOT NULL UNIQUE` | Human-readable. "Lossless", "HD", "Any". |
| `media_type` | `TEXT NOT NULL` | Constrains which rank table is used for evaluation. Profiles are never cross-type. |
| `min_quality_score` | `INTEGER NOT NULL` | Reject releases with quality_score below this. |
| `upgrade_until_score` | `INTEGER NOT NULL` | Stop searching for upgrades once have.quality_score >= this. |
| `min_custom_format_score` | `INTEGER NOT NULL` | Minimum custom format score required. Default 0 (no custom format required). |
| `upgrade_until_format_score` | `INTEGER NOT NULL` | Custom format upgrade ceiling. Default 0. |
| `upgrades_allowed` | `INTEGER NOT NULL` | 0 = never upgrade after initial grab. 1 = upgrade until ceiling. |

**Why INTEGER not UUID for profile IDs:** Quality profiles are created by the admin, not generated by the system. A household will have on the order of 5–20 profiles. Sequential integer IDs are readable in config, in logs, and in foreign keys on `wants`. The low-cardinality argument from RESEARCH.md Open Question 3 applies here.

## Per-type quality rank tables

Each media type has its own rank table. Scores are only compared within a type; a music score of 90 and a video score of 90 mean different things and are never compared. Quality profiles enforce this via the `media_type` column: a profile is applied only against the rank table matching its `media_type`.

### `Music_quality_ranks`

Locked hierarchy from CONTEXT.md: FLAC 24-bit > FLAC 16-bit > ALAC > 320 CBR > V0 VBR > 256 > 128.

```sql

    rank    INTEGER PRIMARY KEY,
    format  TEXT NOT NULL UNIQUE,
    score   INTEGER NOT NULL
);

INSERT INTO music_quality_ranks (rank, format, score) VALUES
    (1, 'FLAC_24BIT',  100),
    (2, 'FLAC_16BIT',   90),
    (3, 'ALAC',          85),
    (4, 'MP3_320_CBR',   70),
    (5, 'MP3_V0_VBR',    65),
    (6, 'MP3_256',       55),
    (7, 'MP3_128',       30);
```

|------|--------|-------|

EPUB tops the hierarchy because it is an open standard with reflowable text and CSS styling support. MOBI and AZW3 are Kindle-specific formats with similar capability but vendor lock-in. PDF is fixed-layout: no reflow, no adaptive font sizing.

### `Comic_quality_ranks`

Comics have their own rank table. CBZ is top: it is the standard open comic archive format. Comics share format names with some book formats (CBR, PDF) but the score assignments differ because the use-case priorities differ.

```sql
CREATE TABLE comic_quality_ranks (
    rank    INTEGER PRIMARY KEY,
    format  TEXT NOT NULL UNIQUE,
    score   INTEGER NOT NULL
);

INSERT INTO comic_quality_ranks (rank, format, score) VALUES
    (1, 'CBZ',  100),
    (2, 'CBR',   90),
    (3, 'PDF',   40);
```

| Rank | Format | Score |
|------|--------|-------|
| 1 | CBZ | 100 |
| 2 | CBR | 90 |
| 3 | PDF | 40 |

CBZ (ZIP-based) is preferred over CBR (RAR-based) because ZIP has no patent history, is universally supported, and CBZ is the de facto standard for comic distribution. Both CBZ and CBR score well above PDF because they preserve page structure and image quality for panel-by-panel reading.

## Evaluation algorithm

Pseudocode for Kritike's assessment logic when a new release is evaluated against an existing have:

```
function evaluate_release(release, want, profile) -> ReleaseDecision:
    // 1. Look up quality score for release's format
    rank_score = query rank table for profile.media_type WHERE format = release.format
    if rank_score is None:
        return REJECT("unknown format")

    release.quality_score = rank_score + release.custom_format_score

    // 2. Check minimum quality floor
    if release.quality_score < profile.min_quality_score:
        return REJECT("below minimum quality score")

    // 3. Check minimum custom format score
    if release.custom_format_score < profile.min_custom_format_score:
        return REJECT("below minimum custom format score")

    // 4. If no existing have, accept
    existing_have = query haves WHERE want_id = want.id ORDER BY quality_score DESC LIMIT 1
    if existing_have is None:
        return ACCEPT("first acquisition")

    // 5. Check whether upgrades are enabled
    if profile.upgrades_allowed = 0:
        return SKIP("upgrades disabled for this profile")

    // 6. Check upgrade ceiling — is existing have already good enough?
    if existing_have.quality_score >= profile.upgrade_until_score:
        return SKIP("upgrade ceiling already met")

    // 7. Check whether this release is actually better
    if release.quality_score <= existing_have.quality_score:
        return SKIP("no quality improvement over existing have")

    return ACCEPT("quality upgrade")
```

**State machine outcome:**
- `ACCEPT("first acquisition")` → trigger download, create have on completion
- `ACCEPT("quality upgrade")` → trigger download, replace have on completion, emit `QualityUpgradeTriggered`
- `REJECT(...)` → discard release, continue searching
- `SKIP(...)` → discard release; if ceiling met, mark want as `fulfilled`

## Custom format scoring

Custom formats add or subtract from the base quality score. They are user-defined rules that match release title patterns and apply a numeric bonus or penalty.

Examples:
- "AMZN WEB-DL" +15 (preferred source)
- "Preferred release group" +50 (trusted encoder)
- "Known bad encode" -100 (always rejected)
- "CAM" -1000 (never acceptable)

**Tables:**

```sql
CREATE TABLE custom_formats (
    id         INTEGER PRIMARY KEY,
    name       TEXT NOT NULL UNIQUE,
    media_type TEXT NOT NULL CHECK(media_type IN (
                   'music', 'audiobook', 'book', 'comic',
                   'podcast', 'movie', 'tv'
               ))
);

CREATE TABLE custom_format_conditions (
    id        INTEGER PRIMARY KEY,
    format_id INTEGER NOT NULL REFERENCES custom_formats(id) ON DELETE CASCADE,
    field     TEXT NOT NULL,
    pattern   TEXT NOT NULL,
    score     INTEGER NOT NULL
);
```

`field` is the release attribute to match: `title`, `indexer`, `release_group`, `codec`, `resolution`, `source`. `pattern` is a regex or exact string. `score` is added to the release's `custom_format_score` when the condition matches.

Multiple conditions on the same custom format are AND-combined (all must match). Multiple custom formats are scored independently and summed.

This is an adaptation of the Sonarr/Radarr custom format model. The scoring mechanism is identical; the field set is extended to cover Harmonia's additional media types.

## Profile scoping rule

Profiles are scoped per `media_type`. A profile named "Lossless" applies only to music; its `min_quality_score` of 85 is evaluated against `music_quality_ranks`, where 85 maps to ALAC. Applying that same score against `video_quality_ranks` would map to UHD Blu-ray H264, a meaningless comparison.

The `quality_profiles.media_type` column enforces this constraint. When Kritike evaluates a release, it queries:

```sql
SELECT * FROM quality_profiles
WHERE id = :profile_id AND media_type = :release_media_type
```

If no row is returned, the profile does not apply to this media type and the release is rejected. Cross-type score comparison never occurs.

See RESEARCH.md Pitfall 3 (Quality Score Drift) for the failure mode this prevents.

## System-wide ownership

Quality profiles are created by the Harmonia administrator, not per household member. All users in a household operate against the same profile set. This matches the architecture assumption that Harmonia is a household-level system, not a multi-tenant one.

Per-user quality overrides, where a household member specifies a different quality ceiling for their requests, are outside this design. If per-user overrides become a requirement, they would be addressed in Phase 7 (SERVE-07 request management) as an extension to the `wants` table, not to `quality_profiles`. Flag this if Aitesis request workflows need per-requester quality preferences.

## Default seed profiles

Harmonia ships with default profiles per media type. Administrators can modify or delete these and add their own.

    ('Lossless', 'music',    85,  100),
    ('Standard', 'music',    70,   90);

-- Audiobook defaults

    ('Any',           'audiobook',  1, 100),

-- Book defaults
INSERT INTO quality_profiles (name, media_type, min_quality_score, upgrade_until_score) VALUES
    ('Any',  'book', 1,   100),
    ('EPUB', 'book', 100, 100);

-- Comic defaults
INSERT INTO quality_profiles (name, media_type, min_quality_score, upgrade_until_score) VALUES
    ('Any', 'comic', 1,   100),
    ('CBZ', 'comic', 100, 100);

-- Podcast defaults
INSERT INTO quality_profiles (name, media_type, min_quality_score, upgrade_until_score) VALUES
    ('Any',          'podcast',  1, 100),
    ('High Quality', 'podcast', 80, 100);

-- Movie defaults
INSERT INTO quality_profiles (name, media_type, min_quality_score, upgrade_until_score) VALUES
    ('Any', 'movie',  1, 100),
    ('HD',  'movie', 60, 100),
    ('UHD', 'movie', 85, 100);

-- TV defaults
INSERT INTO quality_profiles (name, media_type, min_quality_score, upgrade_until_score) VALUES

The "Any" profile (`min_quality_score = 1, upgrade_until_score = 100`) accepts any known format and always upgrades. It is a safe default for items where quality preference is undefined. The named profiles ("Lossless", "HD", "UHD") reflect common household preferences.

## Anti-patterns

| Pattern | What Goes Wrong | Correct Approach |
|---------|----------------|-----------------|
| Cross-type score comparison | Music score 85 (ALAC) compared with video score 85 (UHD Blu-ray H264) | Profiles are scoped by `media_type`; scores only compared within type |
| Single global quality table | One `quality_ranks` table with a `media_type` discriminator column | Separate rank table per media type; no cross-type rows in one table |
| Hardcoded score thresholds | Logic like `if score > 85 then lossless` embedded in code | Scores live in rank tables; thresholds live in profiles; code reads both |
| Per-user profiles in v1 | Household member specifies different quality for each request | System-wide profiles in v1; per-user overrides deferred to Phase 7 |
